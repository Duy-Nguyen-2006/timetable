# Debugging

Active contributors: Duy

This page documents the most common failure modes when running or modifying the Local Agent pipeline and how to diagnose them.

## Agent lifecycle events

Every run of `runLocalAgent` emits a stream of typed `AgentLifecycleEvent` and `AgentEvent` objects via the `onEvent` callback passed in the config.

In the UI these appear as the live step list (thinking → translator → planner → coding → running → checking → fixing).

When debugging, the first thing to do is capture the full event stream (or at least the `stage_started` / `stage_completed` / `error` / `violations_found` events). They almost always tell you exactly which stage failed and why.

## Common failure categories

### 1. Translator produces wrong ConstraintKind or bad params

**Symptoms**
- The agent runs but the resulting schedule obviously violates a constraint the user clearly stated.
- `deterministicReport` shows the constraint was parsed as a different `kind` or with empty/wrong `params`.

**How to debug**
- Look at the raw `translator.rawResponse` (available in test harnesses and via the `onEvent` detail when the stage completes).
- Check whether the fallback parser in `translator.ts` fired (search for the constraint text in the rule-based `parseConstraint` path).
- Compare against `prompts/translator.system.md` — the prompt is the source of truth for how the LLM is supposed to classify the sentence.

**Common causes**
- New constraint kind added to the union but the translator prompt or fallback rules were not updated.
- Ambiguous natural language that matches multiple rules.

### 2. Coder emits Python that fails to compile or crashes at runtime

**Symptoms**
- `execution_result` event with `phase: "compile"` or `status: "crashed"`.
- Python traceback visible in `stderr` or `errorDigest`.

**How to debug**
- The generated code is written to a temp directory by `code_executor.py`. In development you can often find the last `solver_generated.py` in `/tmp/timetable_exec_*`.
- Run the exact generated file manually inside the same Python environment (with the same `input.json`) to reproduce.
- Check whether `skeleton-injector.ts` (AST + syntax checks) passed. If it did, the bug is usually in the prompt or in the skeleton template itself.

**Common causes**
- Prompt change that made the Coder emit code for a kind the current skeleton does not support.
- Missing import or helper function in the skeleton that the Coder assumed existed.

### 3. Sandbox execution times out or is killed

**Symptoms**
- `status: "timeout"` or the process is killed by the container runtime.

**How to debug**
- Look at `durationMs` vs the requested `timeoutMs`.
- In Docker mode, check container logs (`docker logs` or the daemon output).
- In bubblewrap mode, the error is usually a seccomp violation or OOM — the combined stdout/stderr will often say "Killed" or "Bad system call".

**Common causes**
- Solver profile set to `deep` with a very large dataset and no time limit adjustment.
- Accidental quadratic constraint code generated by the Coder (e.g., `for every teacher for every day for every period...` without pruning).

### 4. Deterministic validator reports false positives (or misses real violations)

**Symptoms**
- The schedule looks correct to a human but `hardConstraintPass` is false, or vice-versa.
- `hardCoverageComplete` is false for a constraint you believe should have a checker.

**How to debug**
- Run the exact schedule + input through `validateSchedule` in isolation (there are unit tests that do this).
- Compare the TypeScript checker implementation against the Python one in `validator_engine.py` for the same `kind`.
- Check the `customChecks` array returned by the Python executor — this is where `validator_engine.py` reports per-constraint results.

**Common causes**
- New constraint kind added only on one side (TS or Python).
- Off-by-one in period/day indexing between the two implementations.
- `hardCoverageComplete` logic not updated when a new hard kind was added.

### 5. Token budget exhaustion

**Symptoms**
- `error` event with a message about the token cap.
- Run succeeds for small datasets but fails for realistic ones.

**How to debug**
- The `TokenBudgetGuard` logs cumulative usage per stage.
- Look at which stage consumed the most tokens (usually Translator or Coder on large constraint sets).

**Common causes**
- Very long natural-language constraint lists.
- Previous repair rounds feeding huge `previousAttemptSummary` back into the prompt.

### 6. Round-trip check fails even though the schedule looks valid

**Symptoms**
- `roundTripOk: false` in the validation report even though `hardConstraintPass` is true.

**How to debug**
- The round-trip code (`cp-sat-roundtrip.ts`) re-builds a model that forces the exact schedule the solver returned and asks the solver whether it is still feasible.
- The error message usually names the constraint that became unsatisfiable when the schedule was forced.

This almost always indicates that the generated Python code silently ignored or weakened a constraint (the Coder wrote a comment instead of a real constraint, or used the wrong variable).

## Useful diagnostic commands

```bash
# Run only the Python validator engine tests
./.venv/bin/pytest python/tests/test_validator_engine.py -q

# Run the full TypeScript agent test suite
npx tsx --test "src/features/timetable/ai/local-agent.test.ts"

# Provider smoke (requires key)
npm run provider:smoke

# Inspect the current solver skeleton that the Coder sees
cat public/templates/solver_skeleton.py | head -100
```

## Adding new diagnostics

When you add a new failure mode that is hard to understand from the existing events:

1. Emit a more specific `stage_completed` or `violations_found` event with a `detail` payload.
2. Surface it in the UI (TimetableApp already renders generic `detail`).
3. Add a unit test that reproduces the exact failure and asserts on the events.

Good diagnostics are what allow the Repair stage (and human operators) to actually fix problems instead of just retrying blindly.

See the [AI Pipeline](../systems/ai-pipeline/index.md) pages for the exact contracts of each stage.
